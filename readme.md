# Fe₃O₄–GN Hybrid Nanofluid: Oblique Stagnation-Point Flow Solver + ANN Surrogate

MATLAB project for modeling oblique (non-orthogonal) MHD stagnation-point flow of a
Fe₃O₄–graphene (GN) hybrid nanofluid over a stretching/shrinking sheet, with
variable viscosity and thermal conductivity. Two base fluids are supported:
**water** and **engine oil (EO)**.

The project has two parts:
1. A numerical **BVP solver** (`bvp4c`) that computes the "exact" boundary-layer solution.
2. A set of **ANN surrogate models** (shallow, single-hidden-layer feedforward nets,
   auto-generated by MATLAB's Neural Net Fitting app) trained to reproduce the
   solver's outputs much faster, for use in parameter sweeps / downstream apps.

---

## Physical model

- **Flow**: oblique stagnation-point flow — stream function split into an
  orthogonal part `x·f(η)` and a shear/cross-flow part `g(η)`. `alpha` is the
  stretching/shrinking ratio (shear-to-strain-rate).
- **Effects included**: magnetic field (`M`), exponentially variable viscosity (`m0`),
  exponentially variable thermal conductivity (`n0`), two-step hybrid nanoparticle
  mixing model (`phi1`, `phi2` volume fractions of Fe₃O₄ and GN).
- **Governing ODEs** (`odeSysF`): 7th-order system for
  `f, f', f'', g, g', θ, θ'`, non-dimensionalized via thermophysical ratios
  `B1`–`B5` and Prandtl number `Pr` (computed in `hnf_thermoPhysRatios`).
- **Boundary conditions** (`bcF`): wall (η=0) — `f=0, f'=α, g=1, θ=1`;
  far field (η→∞) — `f'=1, g=0, θ=0`.
- Outputs of interest: **skin friction** `f''(0)`, `g'(0)` and **heat transfer**
  `-θ'(0)`, at the wall.

## Repository structure

| File | Purpose |
|---|---|
| `hnfSysSolve.m` *(main BVP driver)* | Solves the BVP via `bvp4c` for given parameters and both base fluids; produces velocity/temperature profiles, streamlines, and isotherm plots. |
| `dataGenerator.m` | Sweeps `(M, m0, n0, φ, α)` through the solver to generate ANN training data — **water only** (see Known Issues). |
| `numDatagenerator.m` | Same sweep, generates data for **EO only**; used for numerical-vs-ANN comparison plots. |
| `ANN_model.m` | Wrapper that calls the three trained networks below in sequence, applying the critical-α gating logic. |
| `critAlphaModel.m` | ANN — predicts the critical (bifurcation) `α` below which no solution exists, for water and EO. Inputs: `[M m0 n0 φ]`. |
| `skinFrictionModel.m` | ANN — predicts `f''(0)`, `g'(0)` for water and EO. Inputs: `[M m0 n0 φ α]`. |
| `nusseltNoModel.m` | ANN — predicts `-θ'(0)` for water and EO. Inputs: `[M m0 n0 φ α]`. |
| `Plotter.m` | Overlays numerical (`bvp4c`) vs. ANN-predicted curves vs. `α`, for validation. |
| `Training_and_Comparison_data_for_ANN__Fe3O4-_GN_.xlsx` | Raw training/validation data (25k+ rows: skin friction, Nusselt number, critical-α blocks) used to train and validate the three networks. |

## Requirements

- MATLAB with the **Curve Fitting / Deep Learning Toolbox** is *not* required to
  *run* the ANN models — the exported `*.m` files are self-contained (weights are
  hardcoded). It *is* required to retrain them.
- `bvp4c` requires base MATLAB (no toolbox).

---

## Usage

```matlab
% Run the full numerical solver + plots
hnfSysSolve(alpha, n0, m0, M, phi1, phi2, grid_size, max_eta)

% Fast ANN surrogate prediction
params = [M m0 n0 phi alpha];   % single row or Qx5 matrix
res = ANN_model(params);
% res = [M m0 n0 phi alpha  f0_water f0_EO  g0_water g0_EO  theta_water theta_EO]
% NaN entries mean alpha is below the critical (no physical solution exists)
```

## Known issues / TODO

- **Gating logic in `ANN_model.m`** compares a scalar `alpha(i)` against the
  2-element `critAlpha` vector `[water, EO]` in one `if` - this only flags NaN
  when α is below *both* fluids' critical α. If α is between the two critical
  values, the model will still compute (and report) a value for the fluid whose
  solution doesn't actually exist. Gate water/EO independently.

- **ANN accuracy degrades near α ≈ -3 to -3.5** (the strongly-shrinking edge of
  the training sweep), error rising to ~5x the typical mid-range error. Root
  cause: solutions don't exist there for most `(M, m0, n0, φ)` combinations
  (only 14 of 500 combos have any valid point past α=-3), so the network has
  very few training samples to learn that region's curvature near the
  bifurcation. Recommend a finer α step *for those specific combos* near their
  own critical-α boundary, rather than densifying the grid everywhere.

- **`skinFrictionModel`/`nusseltNoModel`/`critAlphaModel` are shallow (single
  hidden layer) networks**, not deep ones - described as "ANN" rather than
  "DNN" for accuracy.
